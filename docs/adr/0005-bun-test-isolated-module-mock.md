# `bun:test` module mocking via a dedicated isolate/restore helper

Unit tests moved from Jest to colocated `bun:test`. `bun:test` has no auto-mocking equivalent to `jest.mock`, so every mocked module uses `mock.module(specifier, factory)` — but `mock.module` replaces a module's exports **process-wide** and mutates them **in place**. Capturing `await import(id)` once and handing that same namespace object back to `mock.module` later to "restore" it does not work, because the captured reference reflects whatever the module was most recently mocked as by the time it's read back — this silently leaks one file's mock into every other colocated test that imports the real module, in a way that depends on file/test execution order (invisible on a normal run, reproducible with `bun test --randomize`).

Fixed by factoring this into `src/test-support/isolated-module-mock.ts`, which captures the real implementation and registers/restores the mock around every individual test (`beforeEach`/`afterEach`, not once per `describe`), rather than re-solving it ad hoc per spec file. `docs/testing.md` documents the pattern and its constraints — in particular, the module specifier must be passed in `src/...`-rooted form, never `./`-relative, since `mock.module`/`import()` inside the helper resolve relative to the helper's own file.

Separately, `jest.resetAllMocks()` (via `bun:test`'s `jest` compat namespace) does **not** reset mocks created with `bun:test`'s own `mock()` — every `beforeEach` calls `.mockReset()` on each named mock instance directly instead of relying on a bulk reset call.
