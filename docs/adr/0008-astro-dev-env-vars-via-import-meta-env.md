# Reading env vars via `process.env`, falling back to `import.meta.env`

`src/utils/env.ts`'s `getEnv()` read `.env` values via raw `process.env.EMAIL_HOST` etc., and `CLAUDE.md` documented this as working "unchanged under Astro" since API routes run in a real Bun process at request time. That assumption was wrong under `astro dev`: Vite's dev server only reliably loads arbitrary `.env` values into `import.meta.env`, not `process.env` — under `astro dev`, `process.env` there just reflects whatever the OS/shell process happened to have. This failed silently (empty strings, no thrown error) rather than loudly, and wasn't fixed by restarting the dev server, since Astro 7's dev server persists as a background daemon across terminal sessions (`astro dev stop`/`status`/`logs` exist precisely because `Ctrl+C` alone doesn't kill it) — so a stale in-memory env from before `.env` existed, or from before a value changed, could persist indefinitely.

The first fix attempt switched `getEnv()` to read `import.meta.env.EMAIL_HOST` etc. exclusively, matching [Astro's documented convention](https://docs.astro.build/en/guides/environment-variables/#getting-environment-variables). That resolved the `astro dev` symptom but broke the *built* server: Vite statically inlines `import.meta.env.*` references into literal values at `astro build` time. `compose.yml`'s `app` service (and the real Cloud Run deployment) inject `EMAIL_HOST`/`EMAIL_USER`/`EMAIL_PASS`/etc. as container **runtime** environment variables/secrets, set after the image was already built — so an `import.meta.env`-only read would freeze those fields to whatever (if anything) was present during the CI build step, silently ignoring the real runtime values. This broke the e2e suite in CI (`Using the form sends the first email...`, `Opening the first email sends the second email`, `Clicking 'Send another' re-shows the email input` — anything depending on an actual send succeeding against the built Docker image), even though it passed locally, because Playwright's local `webServer` runs the built output via a plain `bun ./dist/server/entry.mjs` invocation, where Bun's own automatic `.env` loading happens to populate `process.env` anyway — masking the exact failure mode CI's `docker run`-injected env hits.

Fixed by reading `process.env[key] || import.meta.env[key]` (see the `readEnvVar` helper): `process.env` wins whenever the runtime has actually set it (true for the built server under Docker/Cloud Run, and for any plain `bun`-invoked script thanks to Bun's own `.env` auto-loading), and `import.meta.env` is the fallback for `astro dev`, where `process.env` is empty. `getEnv().dev.isDev` is the one field that stays `import.meta.env.DEV`-only (wrapped in `Boolean(...)`, since Bun's `bun:test` runtime mirrors `import.meta.env` from `process.env` and doesn't synthesize Vite's `DEV`/`PROD`/`MODE` keys, leaving it `undefined` rather than `false` under tests) — unlike the email config, `isDev` *should* stay frozen per build (`true` only for an actual `astro dev` process, `false` for anything built), not vary with runtime env vars. Any new env var read through `getEnv()` needs a matching entry in `src/env.d.ts`'s `ImportMetaEnv` interface, or TypeScript will reject the property access.
