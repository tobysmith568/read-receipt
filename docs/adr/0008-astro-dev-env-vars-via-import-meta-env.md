# Reading env vars via `import.meta.env`, not `process.env`

`src/utils/env.ts`'s `getEnv()` read `.env` values via raw `process.env.EMAIL_HOST` etc., and `CLAUDE.md` documented this as working "unchanged under Astro" since API routes run in a real Bun process at request time. That assumption was wrong under `astro dev`: Vite's dev server only reliably loads arbitrary `.env` values into `import.meta.env`, not `process.env` — under `astro dev`, `process.env` there just reflects whatever the OS/shell process happened to have. This failed silently (empty strings, no thrown error) rather than loudly, and wasn't fixed by restarting the dev server, since Astro 7's dev server persists as a background daemon across terminal sessions (`astro dev stop`/`status`/`logs` exist precisely because `Ctrl+C` alone doesn't kill it) — so a stale in-memory env from before `.env` existed, or from before a value changed, could persist indefinitely.

Fixed by switching `getEnv()` to read `import.meta.env.EMAIL_HOST` etc. instead, matching [Astro's documented convention](https://docs.astro.build/en/guides/environment-variables/#getting-environment-variables). `getEnv().dev.isDev` now comes from Vite's own `import.meta.env.DEV` boolean rather than string-comparing `process.env.NODE_ENV`, wrapped in `Boolean(...)` since Bun's `bun:test` runtime treats `import.meta.env` as a mirror of `process.env` and doesn't synthesize Vite's `DEV`/`PROD`/`MODE` keys, leaving `import.meta.env.DEV` as `undefined` (falsy but not `false`) under tests. Any new env var read through `getEnv()` needs a matching entry in `src/env.d.ts`'s `ImportMetaEnv` interface, or TypeScript will reject the property access.
